# Practice 3. Choosing the Management Method  
## SFedU Student Event Portal

---

# 1. Introduction

The purpose of Practice 3 is to select an appropriate **project management methodology** for the SFedU Student Event Portal.  
The choice should be justified based on the project’s characteristics, constraints, and expected workflow.

This practice builds on the documentation from Practices 1 and 2 and prepares the foundation for scheduling, cost estimation, and structural planning in later practices.

---

# 2. Project Characteristics Relevant to Method Selection

### **Uncertain or evolving requirements**
- Event organizers may request new features as the system develops.  
- User expectations (students, staff) may change after early prototypes.

### **Small academic team**
- Limited time and resources.  
- Need for iterative progress and frequent feedback.

### **Web-based system**
- Suitable for incremental development.  
- Features can be delivered in small, testable parts.

### **Core functionality first**
- Version 1 focuses on essential features.  
- Additional features can be added later.

### **Stakeholder involvement**
- Organizers and supervisors can provide feedback regularly.

These characteristics strongly influence the choice of management method.

---

# 3. Comparison of Management Approaches

## **Waterfall Model**
### Pros:
- clear documentation;  
- structured phases;  
- predictable timeline.

### Cons:
- difficult to adapt to changing requirements;  
- limited flexibility;  
- late user feedback.

## **Agile / Scrum**
### Pros:
- iterative development;  
- early and continuous feedback;  
- flexible scope;  
- suitable for small teams;  
- working product delivered in increments.

### Cons:
- requires regular communication;  
- scope boundaries must be controlled.

## **Kanban**
### Pros:
- simple visual workflow;  
- continuous delivery;  
- flexible task management.

### Cons:
- less structured for academic milestones;  
- not ideal for projects requiring defined increments.

---

# 4. Selected Method: **Agile (Scrum)**

### **Reasoning**
Scrum is the most suitable method for the SFedU Student Event Portal because:

- requirements may evolve as organizers test early versions;  
- the project benefits from short development cycles;  
- the team can deliver core features first, then expand;  
- feedback from students and staff can be incorporated quickly;  
- the academic timeline aligns well with sprint-based planning.

Scrum supports a realistic, iterative approach for a small academic team.

---

# 5. Scrum Framework for This Project

### **Scrum Roles**
- **Product Owner:** academic supervisor or designated team member  
- **Scrum Master:** project manager  
- **Development Team:** students working on frontend, backend, and documentation

### **Scrum Artifacts**
- **Product Backlog:** list of all features (events, registration, tickets, dashboard)  
- **Sprint Backlog:** tasks selected for each sprint  
- **Increment:** working version of the portal after each sprint

### **Scrum Events**
- **Sprint Planning:** define sprint goals  
- **Daily Standups:** short progress sync (optional for academic teams)  
- **Sprint Review:** demonstrate completed features  
- **Sprint Retrospective:** discuss improvements  
- **Sprints:** 1–2 weeks each

---

# 6. Example Sprint Structure

### **Sprint 1**
- event catalog UI  
- basic backend structure  
- database setup

### **Sprint 2**
- event detail pages  
- registration form  
- ticket generation logic

### **Sprint 3**
- organizer dashboard  
- attendee list  
- admin tools

### **Sprint 4**
- polishing UI  
- testing  
- documentation updates

---

# 7. Expected Benefits of Using Scrum

- early visibility of progress;  
- ability to adjust scope as needed;  
- reduced risk of late surprises;  
- continuous improvement;  
- clear structure for academic reporting.

---

# 8. Conclusion

Scrum is selected as the management method for the **SFedU Student Event Portal** due to its flexibility, iterative nature, and suitability for small academic teams.  
This method supports evolving requirements, regular feedback, and incremental delivery — all essential for the successful development of the system.

The next practice will focus on **creating the project schedule and estimating project cost** using the chosen Scrum approach.
